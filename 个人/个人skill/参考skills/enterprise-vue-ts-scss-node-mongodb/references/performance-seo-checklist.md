# 性能与 SEO 检查清单

## 1. 前端性能（Vue）

### 构建与加载

- 启用路由级懒加载与业务模块分包。
- 将重组件（图表、编辑器、地图）做动态导入。
- 控制首屏 JS 体积，优先减少初始依赖。
- 关键资源使用预加载（`preload`）而非全量预取（`prefetch`）。

### 渲染与交互

- 长列表使用虚拟滚动。
- 避免在模板内执行高复杂度计算，转移到 `computed` 或缓存层。
- 避免不必要的深层响应式对象；对大对象做浅层处理。
- 图片使用合适尺寸与格式（WebP/AVIF），并启用延迟加载。

### 预算建议

- LCP < 2.5s
- INP < 200ms
- CLS < 0.1

## 2. Node.js 接口性能

- 统一分页策略，拒绝无上限列表查询。
- 默认返回必要字段，避免大对象“全字段透传”。
- CPU 密集任务放入队列/后台任务，不阻塞主请求。
- 为热点接口增加缓存（本地缓存或分布式缓存）。
- 用日志或 profiling 工具追踪慢接口（按 p95/p99 观察）。

## 3. MongoDB 性能

- 为高频筛选字段建立索引；组合查询建立复合索引。
- 为排序字段配套索引，避免内存排序。
- 所有关键查询使用 `explain('executionStats')` 验证。
- 聚合管道尽早 `$match` 与 `$project`，减少中间数据量。
- 写入前验证幂等与唯一性约束，避免脏写。

## 4. SEO（适用于公开页面）

### 渲染策略

- 有 SEO 诉求页面优先 SSR/SSG（建议 Nuxt）。
- 同一 URL 输出稳定内容，避免客户端二次替换核心正文。

### 页面元信息

- 每页唯一 `title` 与 `meta description`。
- 设置 `canonical`，避免重复内容竞争。
- 补全 Open Graph 与 Twitter Card。
- 提供 `sitemap.xml` 与 `robots.txt`。

### 结构化数据

- 按页面类型加入 JSON-LD（Article、Product、FAQ、Breadcrumb 等）。
- 保证结构化数据与页面可见内容一致。

## 5. 验证建议

- 使用 Lighthouse 对关键页面做前后对比。
- 使用 Search Console 监控抓取、索引和 Core Web Vitals。
- 对关键接口记录基线与优化后耗时（至少 p50/p95）。
- 对关键 Mongo 查询保留 `explain` 结论用于回归对比。
