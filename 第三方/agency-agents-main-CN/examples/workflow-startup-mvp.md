# 多智能体工作流：创业公司MVP

> 一个分步示例，展示如何协调多个智能体从想法到交付MVP。

## 场景

你正在构建一个SaaS MVP——一个面向远程团队的团队回顾工具。你有4周时间交付一个可用的产品，包含用户注册、核心功能和落地页。

## 智能体团队

| 智能体 | 在此工作流中的角色 |
|-------|---------------------|
| 冲刺优先级排序者 | 将项目分解为每周冲刺 |
| UX研究员 | 通过快速用户访谈验证想法 |
| 后端架构师 | 设计API和数据模型 |
| 前端开发者 | 构建React应用 |
| 快速原型师 | 让第一个版本快速运行起来 |
| 增长黑客 | 在构建期间规划发布策略 |
| 现实检查者 | 在进入下一阶段前把关每个里程碑 |

## 工作流

### 第1周：发现 + 架构

**步骤1 — 激活冲刺优先级排序者**

```
激活冲刺优先级排序者。

项目：RetroBoard——面向远程团队的实时团队回顾工具。
时间线：4周到MVP发布。
核心功能：用户认证、创建回顾板、添加卡片、投票、行动项。
约束：独立开发者、React + Node.js技术栈、部署到Vercel + Railway。

将此分解为4个每周冲刺，包含明确的交付物和验收标准。
```

**步骤2 — 激活UX研究员（并行）**

```
激活UX研究员。

我正在构建一个面向远程团队（5-20人）的团队回顾工具。
竞争对手：EasyRetro、Retrium、Parabol。

进行快速竞争分析并识别：
1. 哪些功能是基本要求
2. 竞争对手在哪里不足
3. 我们可以拥有的一个差异化优势

输出一页研究简报。
```

**步骤3 — 交接给后端架构师**

```
激活后端架构师。

这是我们的冲刺计划：[粘贴冲刺优先级排序者输出]
这是我们的研究简报：[粘贴UX研究员输出]

为RetroBoard设计API和数据库模式。
技术栈：Node.js、Express、PostgreSQL、Socket.io用于实时。

交付：
1. 数据库模式（SQL）
2. REST API端点列表
3. 实时看板更新的WebSocket事件
4. 认证策略建议
```

### 第2周：构建核心功能

**步骤4 — 激活前端开发者 + 快速原型师**

```
激活前端开发者。

这是API规范：[粘贴后端架构师输出]

构建RetroBoard React应用：
- 技术栈：React、TypeScript、Tailwind、Socket.io-client
- 页面：登录、仪表板、看板视图
- 组件：RetroCard、VoteButton、ActionItem、BoardColumn

从看板视图开始——这是核心体验。
聚焦实时：当一个用户添加卡片时，所有人都能看到。
```

**步骤5 — 中期现实检查**

```
激活现实检查者。

我们在RetroBoard的4周MVP构建的第2周。

这是我们目前拥有的：
- 数据库模式：[粘贴]
- API端点：[粘贴]
- 前端组件：[粘贴]

评估：
1. 我们能在2周内现实地交付吗？
2. 我们应该削减什么来赶上截止日期？
3. 有什么技术债务会在发布时咬我们？
```

### 第3周：打磨 + 落地页

**步骤6 — 前端开发者继续，增长黑客开始**

```
激活增长黑客。

产品：RetroBoard——团队回顾工具，1周后发布。
目标：远程优先公司的工程经理和Scrum Master。
预算：$0（仅有机发布）。

创建发布计划：
1. 落地页文案（英雄区、功能、CTA）
2. 发布渠道（Product Hunt、Reddit、Hacker News、Twitter）
3. 逐日发布序列
4. 第1周要追踪的指标
```

### 第4周：发布

**步骤7 — 最终现实检查**

```
激活现实检查者。

RetroBoard准备发布。评估生产就绪度：

- 线上URL：[url]
- 测试账户已创建：是
- 错误监控：Sentry已配置
- 数据库备份：每日自动

运行发布检查清单并给出GO / NO-GO决策。
每个标准都需要提供证据。
```

## 关键模式

1. **顺序交接**：每个智能体的输出成为下一个智能体的输入
2. **并行工作**：UX研究员和冲刺优先级排序者可以在第1周同时运行
3. **质量关卡**：中期和发布前的现实检查者防止交付有问题的代码
4. **上下文传递**：始终将前一个智能体的输出粘贴到下一个提示中——智能体之间不共享记忆

## 提示

- 在步骤之间复制粘贴智能体输出——不要总结，使用完整输出
- 如果现实检查者标记了问题，回到相关专家修复它
- 在熟悉手动版本后，可以考虑使用协调者智能体来自动化此流程
