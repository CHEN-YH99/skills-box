# 多智能体工作流：带持久记忆的创业公司MVP

> 与[workflow-startup-mvp.md](workflow-startup-mvp.md)中相同的创业公司MVP工作流，但使用MCP记忆服务器处理智能体之间的状态。不再需要复制粘贴交接。

## 手动交接的问题

在标准工作流中，每个智能体到智能体的转换看起来像这样：

```
激活后端架构师。

这是我们的冲刺计划：[粘贴冲刺优先级排序者输出]
这是我们的研究简报：[粘贴UX研究员输出]

为RetroBoard设计API和数据库模式。
...
```

你就是粘合剂。你在智能体之间复制粘贴输出，跟踪已完成的工作，并希望不会在过程中丢失上下文。这对小项目有效，但在以下情况下会崩溃：

- 会话超时，你丢失了输出
- 多个智能体需要相同的上下文
- QA失败，你需要回退到之前的状态
- 项目跨越数天或数周，涉及多个会话

## 解决方案

安装MCP记忆服务器后，智能体将它们的交付物存储在记忆中，并自动检索所需内容。交接变为：

```
激活后端架构师。

项目：RetroBoard。回忆此项目的先前上下文
并设计API和数据库模式。
```

智能体在记忆中搜索RetroBoard上下文，找到之前智能体存储的冲刺计划和研究简报，并从那里继续。

## 设置

安装任何支持`remember`、`recall`和`rollback`操作的MCP兼容记忆服务器。设置说明见[integrations/mcp-memory/README.md](../integrations/mcp-memory/README.md)。

## 场景

与标准工作流相同：一个SaaS团队回顾工具（RetroBoard），4周到MVP，独立开发者。

## 智能体团队

| 智能体 | 在此工作流中的角色 |
|-------|---------------------|
| 冲刺优先级排序者 | 将项目分解为每周冲刺 |
| UX研究员 | 通过快速用户访谈验证想法 |
| 后端架构师 | 设计API和数据模型 |
| 前端开发者 | 构建React应用 |
| 快速原型师 | 让第一个版本快速运行起来 |
| 增长黑客 | 在构建期间规划发布策略 |
| 现实检查者 | 在进入下一阶段前把关每个里程碑 |

每个智能体的提示中都有一个记忆集成部分（参见[integrations/mcp-memory/README.md](../integrations/mcp-memory/README.md)了解如何添加）。

## 工作流

### 第1周：发现 + 架构

**步骤1 — 激活冲刺优先级排序者**

```
激活冲刺优先级排序者。

项目：RetroBoard——面向远程团队的实时团队回顾工具。
时间线：4周到MVP发布。
核心功能：用户认证、创建回顾板、添加卡片、投票、行动项。
约束：独立开发者、React + Node.js技术栈、部署到Vercel + Railway。

将此分解为4个每周冲刺，包含明确的交付物和验收标准。
完成后记住你的冲刺计划，为此项目打标签。
```

冲刺优先级排序者产出冲刺计划，并以`sprint-prioritizer`、`retroboard`和`sprint-plan`标签存储在记忆中。

**步骤2 — 激活UX研究员（并行）**

```
激活UX研究员。

我正在构建一个面向远程团队（5-20人）的团队回顾工具。
竞争对手：EasyRetro、Retrium、Parabol。

进行快速竞争分析并识别：
1. 哪些功能是基本要求
2. 竞争对手在哪里不足
3. 我们可以拥有的一个差异化优势

输出一页研究简报。完成后记住它，为此项目打标签。
```

UX研究员以`ux-researcher`、`retroboard`和`research-brief`标签存储研究简报。

**步骤3 — 交接给后端架构师**

```
激活后端架构师。

项目：RetroBoard。回忆之前智能体的冲刺计划和研究简报。
技术栈：Node.js、Express、PostgreSQL、Socket.io用于实时。

设计：
1. 数据库模式（SQL）
2. REST API端点列表
3. 实时看板更新的WebSocket事件
4. 认证策略建议

记住每个交付物，为此项目和为frontend-developer打标签。
```

后端架构师自动从记忆中回忆冲刺计划和研究简报。无需复制粘贴。它以`backend-architect`、`retroboard`、`api-spec`和`frontend-developer`标签存储其模式和API规范。

### 第2周：构建核心功能

**步骤4 — 激活前端开发者 + 快速原型师**

```
激活前端开发者。

项目：RetroBoard。回忆后端架构师的API规范和模式。

构建RetroBoard React应用：
- 技术栈：React、TypeScript、Tailwind、Socket.io-client
- 页面：登录、仪表板、看板视图
- 组件：RetroCard、VoteButton、ActionItem、BoardColumn

从看板视图开始——这是核心体验。
聚焦实时：当一个用户添加卡片时，所有人都能看到。
记住你的进度，为此项目打标签。
```

前端开发者从记忆中拉取API规范并据此构建。

**步骤5 — 中期现实检查**

```
激活现实检查者。

项目：RetroBoard。我们在4周MVP构建的第2周。

回忆此项目之前智能体的所有交付物。

评估：
1. 我们能在2周内现实地交付吗？
2. 我们应该削减什么来赶上截止日期？
3. 有什么技术债务会在发布时咬我们？

记住你的裁决，为此项目打标签。
```

现实检查者对目前产出的所有内容拥有完整可见性——冲刺计划、研究简报、模式、API规范和前端进度——无需你收集和粘贴所有内容。

### 第3周：打磨 + 落地页

**步骤6 — 前端开发者继续，增长黑客开始**

```
激活增长黑客。

产品：RetroBoard——团队回顾工具，1周后发布。
目标：远程优先公司的工程经理和Scrum Master。
预算：$0（仅有机发布）。

回忆项目上下文和现实检查者的裁决。

创建发布计划：
1. 落地页文案（英雄区、功能、CTA）
2. 发布渠道（Product Hunt、Reddit、Hacker News、Twitter）
3. 逐日发布序列
4. 第1周要追踪的指标

记住发布计划，为此项目打标签。
```

### 第4周：发布

**步骤7 — 最终现实检查**

```
激活现实检查者。

项目：RetroBoard，准备发布。

回忆所有项目上下文、之前的裁决和发布计划。

评估生产就绪度：
- 线上URL：[url]
- 测试账户已创建：是
- 错误监控：Sentry已配置
- 数据库备份：每日自动

运行发布检查清单并给出GO / NO-GO决策。
每个标准都需要提供证据。
```

### 当QA失败时：回滚

在标准工作流中，当现实检查者拒绝一个交付物时，你回到负责的智能体并试图解释出了什么问题。有了记忆，恢复循环更紧密：

```
激活后端架构师。

项目：RetroBoard。现实检查者标记了API设计的问题。
回忆现实检查者的反馈和你之前的API规范。
回滚到你最后已知的良好模式，并解决提出的具体问题。
完成后记住更新后的交付物。
```

后端架构师可以确切地看到现实检查者标记了什么，回忆自己之前的工作，回滚到检查点，并产出修复——所有这些都不需要你手动跟踪版本。

## 前后对比

| 方面 | 标准工作流 | 带记忆的工作流 |
|------|-----------|---------------|
| **交接** | 在智能体之间复制粘贴完整输出 | 智能体自动回忆所需内容 |
| **上下文丢失** | 会话超时丢失一切 | 记忆跨会话持久化 |
| **多智能体上下文** | 手动从N个智能体编译上下文 | 智能体搜索项目标签的记忆 |
| **QA失败恢复** | 手动描述出了什么问题 | 智能体回忆反馈 + 回滚 |
| **多日项目** | 每个会话重新建立上下文 | 智能体从上次离开处继续 |
| **所需设置** | 无 | 安装MCP记忆服务器 |

## 关键模式

1. **用项目名称标记一切**：这是让回忆工作的关键。每条记忆都用`retroboard`（或你的项目名称）打标签。
2. **为接收智能体标记交付物**：当后端架构师完成API规范时，它用`frontend-developer`标记记忆，以便前端开发者在回忆时找到它。
3. **现实检查者获得完整可见性**：因为所有智能体将其工作存储在记忆中，现实检查者可以回忆项目的所有内容，无需你编译。
4. **回滚替代手动撤销**：当某事失败时，回滚到上一个检查点，而不是试图弄清楚什么改变了。

## 提示

- 你不需要一次修改每个智能体。从为你最常用的智能体添加记忆集成开始，然后逐步扩展。
- 记忆指令是提示，不是代码。LLM解释它们并根据需要调用MCP工具。你可以调整措辞以匹配你的风格。
- 任何支持`remember`、`recall`、`rollback`和`search`工具的MCP兼容记忆服务器都可以与此工作流配合使用。
