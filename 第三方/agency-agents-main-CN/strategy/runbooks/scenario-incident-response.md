# 🚨 运行手册：事件响应

> **模式**：NEXUS-Micro | **持续时间**：数分钟至数小时 | **智能体**：3-8

---

## 场景

生产环境出了问题。用户受到影响。响应速度很重要，但正确处理同样重要。本运行手册涵盖从检测到事后复盘的全过程。

## 严重程度分类

| 级别 | 定义 | 示例 | 响应时间 |
|------|------|------|----------|
| **P0 — 关键** | 服务完全中断、数据丢失、安全漏洞 | 数据库损坏、DDoS 攻击、认证系统故障 | 立即（全员响应） |
| **P1 — 高** | 主要功能损坏、严重性能下降 | 支付处理中断、50%+ 错误率、10 倍延迟 | < 1 小时 |
| **P2 — 中** | 次要功能损坏、有替代方案 | 搜索不工作、非关键 API 错误 | < 4 小时 |
| **P3 — 低** | 外观问题、轻微不便 | 样式错误、错别字、轻微 UI 故障 | 下一个冲刺 |

## 按严重程度的响应团队

### P0 — 关键响应团队
| 智能体 | 角色 | 行动 |
|-------|------|------|
| **Infrastructure Maintainer** | 事件指挥官 | 评估范围，协调响应 |
| **DevOps Automator** | 部署/回滚 | 如需要则执行回滚 |
| **Backend Architect** | 根因调查 | 诊断系统问题 |
| **Frontend Developer** | 客户端调查 | 诊断客户端问题 |
| **Support Responder** | 用户沟通 | 状态页面更新，用户通知 |
| **Executive Summary Generator** | 利益相关者沟通 | 实时高管更新 |

### P1 — 高响应团队
| 智能体 | 角色 |
|-------|------|
| **Infrastructure Maintainer** | 事件指挥官 |
| **DevOps Automator** | 部署支持 |
| **相关开发者智能体** | 修复实现 |
| **Support Responder** | 用户沟通 |

### P2 — 中响应
| 智能体 | 角色 |
|-------|------|
| **相关开发者智能体** | 修复实现 |
| **Evidence Collector** | 验证修复 |

### P3 — 低响应
| 智能体 | 角色 |
|-------|------|
| **Sprint Prioritizer** | 添加到待办列表 |

## 事件响应序列

### 步骤 1：检测与分诊（0-5 分钟）

```
触发：来自监控的告警 / 用户报告 / 智能体检测

Infrastructure Maintainer：
1. 确认告警
2. 评估范围和影响
   - 多少用户受影响？
   - 哪些服务受影响？
   - 数据是否有风险？
3. 分类严重程度（P0/P1/P2/P3）
4. 激活适当的响应团队
5. 创建事件频道/线程

输出：事件分类 + 响应团队已激活
```

### 步骤 2：调查（5-30 分钟）

```
并行调查：

Infrastructure Maintainer：
├── 检查系统指标（CPU、内存、网络、磁盘）
├── 审查错误日志
├── 检查最近部署
└── 验证外部依赖

Backend Architect（如果是 P0/P1）：
├── 检查数据库健康状态
├── 审查 API 错误率
├── 检查服务间通信
└── 识别故障组件

DevOps Automator：
├── 审查最近的部署历史
├── 检查 CI/CD 管道状态
├── 如需要则准备回滚
└── 验证基础设施状态

输出：根因已识别（或缩小到组件级别）
```

### 步骤 3：缓解（15-60 分钟）

```
决策树：

IF 由最近部署引起：
  → DevOps Automator：执行回滚
  → Infrastructure Maintainer：验证恢复
  → Evidence Collector：确认修复

IF 由基础设施问题引起：
  → Infrastructure Maintainer：扩展/重启/故障转移
  → DevOps Automator：支持基础设施变更
  → 验证恢复

IF 由代码缺陷引起：
  → 相关开发者智能体：实施热修复
  → Evidence Collector：验证修复
  → DevOps Automator：部署热修复
  → Infrastructure Maintainer：监控恢复

IF 由外部依赖引起：
  → Infrastructure Maintainer：激活备用/缓存
  → Support Responder：向用户沟通
  → 监控外部恢复

全程：
  → Support Responder：每 15 分钟更新状态页面
  → Executive Summary Generator：简报利益相关者（仅 P0）
```

### 步骤 4：解决验证（修复后）

```
Evidence Collector：
1. 验证修复解决了问题
2. 工作状态的截图证据
3. 确认未引入新问题

Infrastructure Maintainer：
1. 验证所有指标恢复正常
2. 确认没有级联故障
3. 修复后监控 30 分钟

API Tester（如果与 API 相关）：
1. 对受影响端点运行回归测试
2. 验证响应时间已正常化
3. 确认错误率回到基线

输出：事件解决确认
```

### 步骤 5：事后复盘（48 小时内）

```
Workflow Optimizer 主导事后复盘：

1. 时间线重建
   - 问题何时引入？
   - 何时被检测到？
   - 何时被解决？
   - 用户受影响的总时长

2. 根因分析
   - 什么出了故障？
   - 为什么会出故障？
   - 为什么没有更早发现？
   - 5 个为什么分析

3. 影响评估
   - 受影响用户数
   - 收入影响
   - 声誉影响
   - 数据影响

4. 预防措施
   - 什么监控能更早发现此问题？
   - 什么测试能防止此问题？
   - 需要什么流程变更？
   - 需要什么基础设施变更？

5. 行动项
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]

输出：事后复盘报告 → Sprint Prioritizer 将预防任务添加到待办列表
```

## 沟通模板

### 状态页面更新（Support Responder）
```
[时间戳] — [服务名称] 事件

状态：[调查中 / 已识别 / 监控中 / 已解决]
影响：[用户影响描述]
当前行动：[我们正在采取的措施]
下次更新：[预计下次更新的时间]
```

### 高管更新（Executive Summary Generator — 仅 P0）
```
事件简报 — [时间戳]

情况：[服务] [中断/降级]，影响 [N 用户/流量百分比]
原因：[已知/调查中] — [简要描述（如已知）]
行动：[正在采取的措施] — 预计完成时间 [时间估计]
影响：[业务影响 — 收入、用户、声誉]
下次更新：[时间戳]
```

## 升级矩阵

| 条件 | 升级给 | 行动 |
|------|--------|------|
| P0 在 30 分钟内未解决 | Studio Producer | 增加资源，供应商升级 |
| P1 在 2 小时内未解决 | Project Shepherd | 资源重新分配 |
| 疑似数据泄露 | Legal Compliance Checker | 监管通知评估 |
| 用户数据受影响 | Legal Compliance Checker + Executive Summary Generator | GDPR/CCPA 通知 |
| 收入影响 > $X | Finance Tracker + Studio Producer | 业务影响评估 |
